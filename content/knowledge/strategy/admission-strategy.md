## 一、什么是准入策略？

当一个人申请贷款时，金融机构需要在短时间内做出一个决策：**批准还是拒绝？批多少额度？**

**准入策略**就是把这个决策过程标准化、系统化的规则集合。

一套好的准入策略需要做到：
- 拒绝那些大概率会违约的申请人
- 批准那些大概率会正常还款的申请人
- 对于中间地带的申请人，用额度、利率等工具管理风险

---

## 二、准入策略的层次结构

一套完整的准入策略通常是"漏斗型"的，申请人需要逐层通过：

```
所有申请进件
    ↓ 第一层
硬拒绝规则（Hard Reject）
──────────────────────────
命中即拒绝，不进入后续流程
    ↓ 通过
    ↓ 第二层
反欺诈规则
──────────────────────────
识别欺诈风险，高风险直接拒绝
    ↓ 通过
    ↓ 第三层
信用评分分层
──────────────────────────
按 A 卡分数决定：通过/复核/拒绝
    ↓ 通过
额度与利率定价
──────────────────────────
根据风险水平确定具体额度和利率
    ↓
放款
```

## 三、第一层：硬拒绝规则

### 什么是硬规则？

硬规则（Hard Rules）是"**只要命中，就立刻拒绝**"的条件，不需要看评分，不需要人工审核。

硬规则针对的是**客观上不适合放贷**的情况：

| 硬规则类型 | 具体示例 | 设置原因 |
|-----------|----------|----------|
| 年龄限制 | 年龄 < 18 或 > 70 | 法律约束（未成年）或还款能力显著下降 |
| 严重征信问题 | 当前有未结清逾期 | 已经还不起了，再借只会更坏 |
| 历史严重违约 | 征信有呆账/核销记录 | 有主观不还的历史记录 |
| 司法失信 | 命中失信被执行人名单（老赖） | 主观拒绝还款的最强信号 |
| 欺诈黑名单 | 历史有欺诈记录 | 有借钱不还的前科 |
| 产品不匹配 | 申请金额远超收入能力 | 明显超出还款能力 |

### 硬规则的设计原则

**宁少勿多，宁松勿紧。**

硬规则一旦命中，申请人永远没有机会被重新审视。如果把条件设得太苛刻，会误杀大量实际上能正常还款的好客户。

**错误示范**：把"近 2 年内有任何逾期"设为硬规则  
→ 很多人可能有一次 3 天的偶发逾期，但整体信用良好，这样会误杀大量好客户

**正确示范**：把"当前有未结清逾期"或"征信有核销记录"设为硬规则  
→ 这类情况下的风险是客观的，误杀概率很低

## 四、第二层：反欺诈规则

### 欺诈风险 vs 信用风险

| 维度 | 信用风险 | 欺诈风险 |
|------|----------|----------|
| 本质 | 真实用户，无力或不愿还款 | 假冒他人或提供虚假信息骗贷 |
| 识别方法 | 信用模型（概率预测） | 异常检测（规则 + 模型） |
| 处置方式 | 调整额度/利率管理风险 | 直接拒绝，上报 |

欺诈者可能伪造完美的征信记录，A 卡会给他高分。所以**反欺诈必须在信用评分之前**独立完成。

### 常见反欺诈规则

**设备与网络维度**

| 规则 | 说明 |
|------|------|
| 同一设备指纹，近 24 小时内有 ≥ 3 个不同身份申请 | 一台设备用多个身份申请 = 团伙欺诈信号 |
| IP 地址来自高风险代理/VPN | 隐藏真实位置，规避风控 |
| 同一 IP，近 1 小时内 ≥ 5 次申请 | 批量欺诈攻击信号 |

**申请行为维度**

| 规则 | 说明 |
|------|------|
| 表单填写时间 < 10 秒 | 正常人不可能这么快，机器人或复制粘贴 |
| 关键字段（身份证号、手机号）使用粘贴操作 | 不是手动输入，可能是从别处复制 |
| 同一时间多个申请行为同时进行 | 批量操作信号 |

**信息一致性维度**

| 规则 | 说明 |
|------|------|
| 手机号与身份证归属地相差极大（如身份证青海，手机号北京，但声称本地居住） | 信息矛盾 |
| 银行卡开户行与身份信息不匹配 | 可能使用他人银行卡 |
| 人脸比对通过率 < 阈值 | 活体或人脸与身份证不符 |

### 反欺诈模型

除了规则，还可以使用反欺诈模型：
- **风险评分**：输出一个"欺诈概率"，高于阈值触发人工审核或拒绝
- **关联图谱**：识别申请人与已知欺诈团伙的关系网络（手机号、设备、银行卡、住址重叠等）

## 五、第三层：信用评分分层

通过前两层后，进入评分卡打分阶段。

### 分层策略设计

按 A 卡分数设置不同的处置策略：

| 分数段 | 风险等级 | 策略 | 额度政策 |
|--------|----------|------|----------|
| ≥ 700 分 | 极低风险 | 自动通过 | 最高额度 |
| 660–699 分 | 低风险 | 自动通过 | 标准额度 |
| 620–659 分 | 中等风险 | 条件通过或人工复核 | 较低额度 |
| 580–619 分 | 较高风险 | 人工复核或小额尝试 | 最低额度，或要求补充材料 |
| < 580 分 | 高风险 | 自动拒绝 | — |

**灰色区间的处理：**

中间分数段（620–659 分）是风控最有挑战性的地方：
- 如果直接拒绝，会损失大量实际能正常还款的客户
- 如果直接通过，坏账率会上升

常见处理方式：
1. **人工复核**：补充材料（银行流水、工资证明），人工判断
2. **小额尝试**：给极低额度（如 1000 元），观察首期还款行为后再决定是否增额
3. **附加条件**：要求提供担保人或增信手段

### 多维度分层

评分卡不是唯一的分层维度。可以结合多个维度进行精细化分层：

**举例：A 卡分 × 多头借贷维度的二维矩阵**

| | 无多头借贷 | 多头借贷1–2家 | 多头借贷3家以上 |
|--|-----------|--------------|----------------|
| **A卡高分** | 自动通过，高额度 | 自动通过，标准额度 | 人工复核 |
| **A卡中分** | 自动通过，较低额度 | 人工复核 | 拒绝 |
| **A卡低分** | 小额尝试 | 拒绝 | 拒绝 |

这比单一维度的分层更精细，但也需要更多的策略维护工作。

## 六、策略调整的量化分析

准入策略不是一次设定好就不变的，需要随着市场变化、模型迭代持续调整。每次调整都需要**严格的量化分析**。

### 6.1 回溯分析（Backtest）

在上线新策略之前，用历史数据模拟：

```
新策略通过量 = 历史申请中，按新策略判定通过的数量
新策略坏账预估 = 新增通过客群的 A 卡分 → 对应的历史坏账率

影响评估：
- 通过率变化 = 新通过量 / 老通过量 - 1
- 坏账率变化 = 新加进来的客户坏账率 - 原有客户坏账率
- 净收益变化 = 新增通过量 × 平均贷款额 × 利率 - 新增通过量 × 坏账率 × LGD × 贷款额
```

**回溯分析的局限性**：

回溯只能告诉你"如果过去按这个策略做会怎样"，但无法预测未来的市场变化和客群变化。

### 6.2 AB 测试

回溯通过后，在线上做小流量实验验证：

**实验设计：**
- 实验组（10–20% 流量）：按新策略执行
- 对照组（80–90% 流量）：按老策略执行
- 实验时长：至少 3–6 个月，确保有足够的样本和观察期

**关注指标：**
- 通过率：是否达到预期变化
- 首逾率：首期即逾期的比例
- M1 逾期率：30 天逾期率（需要等 1–2 个月表现期）
- 净收益：收入增加 vs 坏账损失增加

**统计显著性检验：**

两组差异需要达到统计显著性（通常 p < 0.05），排除随机波动的干扰，才能确认新策略有效。

### 6.3 策略调整的触发条件

| 触发信号 | 调整方向 |
|----------|----------|
| 首逾率显著上升 | 收紧准入（提高分数阈值） |
| 通过率过低，好客户大量流失 | 适度放松（降低分数阈值） |
| 特定渠道坏账率显著高于平均 | 对该渠道单独收紧 |
| 模型 PSI > 0.2，客群漂移 | 重新评估分数切割点 |
| 宏观经济下行 | 全面收紧，或增加高风险客群的人工审核 |

## 七、准入策略常见误区

**误区1：策略越严格越安全**

错。策略过严会导致大量好客户被拒绝，损失收入；同时被拒客户可能流向竞争对手，提高竞争对手的质量，反而在竞争中处于不利。

**误区2：评分卡是万能的，规则没必要**

错。规则处理边界情况，评分卡处理连续分布。极端情况（如已有核销记录、在逃嫌疑人）不需要用模型，规则直接处理更高效。

**误区3：策略调整不需要 AB 测试**

危险。即使回溯结果很好，真实上线后也可能因为客群偏移、外部环境变化而表现不同。AB 测试是唯一可靠的上线验证方式。

**误区4：策略一旦上线就不需要维护**

错。准入策略需要根据业务目标、客群变化、宏观环境持续调整，通常每季度都需要做策略回顾。

---

## 小结

准入策略是贷前风控的"最终决策层"，好的准入策略需要：

1. **层次分明**：欺诈 → 硬规则 → 评分分层，各司其职
2. **精准而不过度**：硬规则不过紧，评分切割点有数据支撑
3. **持续迭代**：随市场变化调整，上线前 Backtest，上线后 AB Test
4. **量化驱动**：每次调整必须能量化影响，用数据说话
